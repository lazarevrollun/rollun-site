# Epic 7 Context: CMS — паспорт компании и управляемый контент страниц (Фаза 2)

<!-- Generated from planning artifacts. Regenerate with compile-epic-context if planning docs change. -->

## Goal

Дать нетехническому менеджеру возможность править паспорт компании, изображения и тексты страниц через админку Payload — без разработчика, без пересборки и без риска сломать пиксель-в-пиксель вёрстку. Эпик переводит захардкоженный контент Фазы 1 (Epic 1/6) на управляемые типизированные слоты Payload CMS: паспорт компании (`SiteSettings`), медиа с авто-оптимизацией (`Media`), контент страниц (Page Globals) и on-demand ревалидацию на проде. Разметка и визуал не меняются — меняется только источник контента. Товары/бренды/магазины из CMS остаются на Epic 8 (Фаза 3).

## Stories

- Story 7.1: Паспорт компании — SiteSettings
- Story 7.2: Медиа — коллекция Media и оптимизация изображений
- Story 7.3: Управляемый контент страниц — Page Globals
- Story 7.4: Ревалидация контента on-demand

## Requirements & Constraints

- Правка любого паспорт-атома (телефон, адрес, email, соц-ссылка, часы) в одном месте обновляет все места вывода: шапку, футер, все страницы. Ни один атом не должен иметь второго владельца.
- Менеджер меняет размеченные слоты (картинки, тексты) через админку; изменения появляются на проде за секунды без пересборки и без правок кода.
- Свобода редактирования строго ограничена типизированными слотами: менеджер физически не может изменить вёрстку/структуру (контр-цель — не возвращать риск «менеджер сломал дизайн»). Свободный page-builder и абстрактная модель `Page` запрещены.
- Изображения оптимизируются автоматически (webp, нужные размеры); `Media` — единственный владелец канонических размеров/форматов.
- Успех эпика: admin/manager самостоятельно меняет картинку/текст/паспорт, результат виден на проде без пересборки; при этом сохранён пиксель-в-пиксель по Handoff (desktop+mobile).
- Роли `admin`/`manager` ограничивают доступ через встроенный access control Payload. Конкретный матрикс прав — TBD (assumption), детализируется на Фазе 3/Epic 8.

## Technical Decisions

- **Паспорт — единственный владелец атома (AD-14).** Каждый паспорт-атом имеет единственный дом в `SiteSettings`; страницы/shell/footer ссылаются, не дублируют. Кросс-страничные расхождения дизайна — это разные именованные атомы (`hours.store` на Our Shops ≠ `hours.homeCta`; `address.houston` — общий для Our Shops и Contact), а НЕ одно поле, отрендеренное по-разному. Не сводить расхождения к одному значению.
- **Один Global на страницу, без page-builder (AD-5).** Globals: `HomeContent`, `AboutContent`, `CatalogContent`, `BrandsContent`, `ShopsContent`, `ContactContent` + `SiteSettings`. Слот = типизированное поле под маркер захардкоженной вёрстки. `BrandsContent` (страница собственного бренда MOTOTOU) ≠ коллекция `Brands` (лого чужих брендов для marquee Catalog) — разные сущности.
- **Граница код/контент по 4 уровням текучести (AD-6).** 🔴 живое (картинки, товары) и 🟡 правимый текст живых блоков → CMS; 🟢 паспорт → `SiteSettings`; ⚫ гвоздями (вёрстка, структура, юр/статичный текст, микрокопи) → код. Tie-break на границе 🟡↔⚫: по умолчанию ⚫ (поле остаётся в коде); слотом становится только явным промоушеном. При сомнении — гвоздями.
- **Страница = чистая функция типизированного контент-объекта (AD-7).** Фаза 1 подаёт статический инстанс типа (`content/*`); Фаза 2 подаёт тот же тип из Payload Global. Меняется источник, не разметка → пиксель между фазами не дрейфует. TS-тип контент-объекта = Payload-генерённый тип этого Global (или typecheck-совместимое подмножество); совместимость типов проверяется в CI. Переключение хардкода Epic 1/6 на ссылки — без изменения разметки.
- **Ревалидация on-demand (AD-10, NFR-5).** Страницы рендерятся статически; правка Global триггерит `revalidateTag` (НЕ `revalidatePath`) из `afterChange`-хука Payload. Каждый Global имеет канонический тег; паспорт/shell тегируют ВСЕ поверхности (mobile+desktop), иначе desktop/mobile одной страницы разойдутся в свежести. Per-page ad-hoc кэша нет.
- **Модель кэша Next 16.** Согласовано с `cacheComponents: true` + `use cache` + `cacheTag`. Без `cacheTag` `revalidateTag` не сработает — тегирование обязательно.
- **Изображения (NFR-4).** Вывод везде через `next/image`; `Media` (webp, нужные размеры) — единственный владелец канонических размеров/форматов.
- **Один рантайм (AD-12).** Один процесс Next.js 16 (Node 20+) отдаёт публичный сайт + Payload-админку (`/admin`) + local API. Контент читается через Payload local API (`getPayload`) на request/build-времени — сетевого хопа к CMS из браузера нет (AD-4). Env: DB-коннекшн, `PAYLOAD_SECRET`, revalidate-секрет. DB-адаптер — Postgres (assumption, владельцу подтвердить). Версия Payload — последний стабильный 3.x (не 4.0).
- **Раскладка (conventions).** `globals/` — `SiteSettings`, `<Page>Content`; `collections/` — `Media` (и позже Products/Brands/Shops/Posts/Users); `content/*` — статические инстансы Фазы 1; `lib/payload.ts` — `getPayload`.

## UX & Interaction Patterns

- **Art-direction сохраняется при переезде на Media (AD-3).** На каждом вьюпорте грузится ровно одна композиция изображений (тяжёлые hero 15–18 МБ нельзя тянуть в оба дерева). Обе композиции desktop|mobile SSR-рендерятся, переключение только CSS-медиа на 768px. Переезд hero на `Media`/`next/image` (`sizes`) не должен ломать этот принцип.
- **Известные дефекты дизайна воспроизводятся дословно (AD-13).** Расхождения часов/адресов между страницами — намеренно разные атомы, их не примиряют. Переезд паспорта в `SiteSettings` сохраняет эти расхождения через разные именованные поля.

## Cross-Story Dependencies

- Зависит от Epic 1 (shell/footer, layout, DS-токены) и Epic 6 (Our Shops, Contact — источники хардкоженного паспорта): Story 7.1 переключает этот хардкод на ссылки `SiteSettings`.
- Внутри эпика: 7.1 (`SiteSettings`) и 7.2 (`Media`) — фундамент для 7.3 (Page Globals ссылаются на паспорт-атомы и медиа); 7.4 (ревалидация) завершает контур для всех Globals и паспорта.
- Story 7.3 требует совместимости Payload-генерённых типов с контрактами страниц Фазы 1 (AD-7) — общий контракт TS-типа между RSC, островком и Payload Global.
- Epic 8 (Фаза 3) наследует контент-модель: коллекции `Products`/`Brands`/`Shops` и матрикс ролей строятся поверх этого фундамента; `Media` (7.2) переиспользуется товарными изображениями.
