## Покрыто

Практически весь брейншторм перенесён, местами обогащён:

- **Стек и ключевой ход** — Next.js App Router + TS + React + Payload CMS 3.0 в одном процессе, local API без сетевого хопа → PRD §1, addendum §A. «Захардкоженный layout + типизированные слоты» (снимает три напряжения) → PRD §1, §4.9, addendum §A.
- **Модель контента** — `SiteSettings` (паспорт) + по одному Global на страницу (`HomeContent`/`AboutContent`/`CatalogContent`/`BrandsContent`/`ShopsContent`/`ContactContent`); коллекции `Products`(sku-шов)/`Brands`/`Shops`/`Media`/`Posts`/`Users` → addendum §C, PRD глоссарий §3.
- **sku/externalId-шов** под будущий автосинк (цена/картинка/наличие поверх) → PRD §4.4, addendum §C, §E.
- **Уровни текучести контента** (🔴🟡🟢⚫) и граница «код vs контент» → addendum §C.
- **Дизайн-фиделити** — DS-токены 1:1 из `colors_and_type.css`, оранж `#EA7B08`, шрифты Archivo/Hanken Grotesk/Spline Sans Mono, 6 desktop + mobile, «матчим визуал, не DOM» → PRD §1, §3, §4.1–4.7, SM-1; addendum §B. **Обогащено:** addendum §B добавил вторую (mobile) DS — Poppins/Roboto/Karla, оранж `#EF7F1A` — чего в брейншторме не было (это уточнение из handoff, не пропуск).
- **Media/next/image авто-оптимизация** («страх hero на 8 МБ») → addendum §A, §C.
- **On-demand revalidation / ISR** → PRD FR-13, addendum §A.
- **Роадмап Фазы 0–4** (дизайн 1:1 первым, CMS слоями потом) → PRD §4.9, §6, addendum §D.
- **Явные scope-решения** (автосинк не сейчас, ручной ввод, сайт некритичный) → PRD §5, §6.2.
- **Автономный разворот приоритетов** (дизайн №1, CMS вторична) — legitimately отражён, соответствует memlog-решению пользователя.

## Пропущено

Материальных архитектурных пропусков не выявлено — PRD/addendum полнее брейншторма. Ниже мелочи, которые downstream (архитектура/UX/dev) может захотеть иметь явными:

1. **Точный путь к handoff-прототипам.** Брейншторм указывает `rollun_handoff/rollun-web-site/project/` (подкаталог `project/` — где лежат HTML-прототипы). PRD §3 (глоссарий) сокращает до `rollun_handoff/rollun-web-site/`. Полный путь восстановлен только в addendum §H. Для агента, который будет физически открывать прототипы, дубль-указание в основном нарративе PRD было бы надёжнее. (Скорее «искажено», см. ниже.)

2. **Явная связка «поля автосинка».** Брейншторм формулирует шов как «дописывал **цену/картинку/наличие** поверх». Addendum §C это сохраняет, но в самом PRD (FR-4/FR-5, где перечислены поля товара из дизайна: brand/domain/name/images/rating/specs/fits/offers) поля автосинка не названы — downstream-архитектору полезно знать, какие именно поля модель `Products` должна оставить «перезаписываемыми фидом». Не потеряно (есть в addendum), но разнесено.

3. **Первичная цель брейншторма как формулировка.** memlog `goal`: «Менеджеры сами меняют картинки, тексты, **даты работы**, посты — без правки кода». Все объекты покрыты (часы → `SiteSettings`, посты → `Posts` Фаза 4), но исходная формулировка «менеджер-центричной» цели в PRD-видении сознательно понижена до вторичной (следствие autonomous-разворота — легитимно).

Ничего из перечисленного не является потерей архитектурного решения или связи модели данных.

## Искажено

1. **Путь к handoff** (см. Пропущено #1): PRD §3 указывает корень бандла без подкаталога `project/`, где реально лежат HTML. Разночтение между §3 и addendum §H. Рекомендация: выровнять на `rollun_handoff/rollun-web-site/project/`.

2. **Приоритет проекта переставлен относительно исходной цели брейншторма** — но это НЕ искажение: пользователь явно сделал autonomous-разворот (memlog: «приоритет №1 — дизайн пиксель-в-пиксель. CMS/админка вторичны»). PRD корректно следует последней воле пользователя, а не первоначальной frontmatter-цели.

3. **Форма → CRM API** — по условию задачи это осознанное изменение пользователя, не искажение. Отражено консистентно во всех разделах PRD и addendum (нет `Submissions`, нет mailer, POST в CRM, endpoint через env-переменную, контракт TBD).

## Вывод

Перенос брейншторма в PRD/addendum — практически полный и местами обогащающий (вторая DS, постраничные адреса, канонизация CRM-развязки). Материальных пропусков архитектуры или связей модели данных нет. Единственная предметная рекомендация — выровнять путь к handoff (`.../project/`) между PRD §3 и addendum §H, чтобы сборочный агент не искал прототипы не там.
